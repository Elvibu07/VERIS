1. Arquitectura y Paradigma del Sistema

La plataforma VERIS está diseñada bajo un paradigma de Cliente Robusto sin Servidor Dedicado. Esto significa que el 100% de la lógica de negocio, validaciones de seguridad, enrutamiento y operaciones de persistencia de datos ocurren directamente en el navegador del usuario utilizando JavaScript (ES6+) y la API de LocalStorage.

Nota Arquitectónica: El uso de LocalStorage como "Base de Datos simulada" permite que la plataforma opere instantáneamente sin latencia de red, haciéndola perfecta para demostraciones, prototipados rápidos y despliegues estáticos en CDNs como Vercel o GitHub Pages.

2. Matriz de Perfiles y Funcionalidades

La plataforma soporta 3 perfiles de acceso. Al iniciar sesión en index.html, el script auth.js evalúa el atributo rol del usuario y restringe el acceso a las vistas correspondientes.

2.1. 👨‍💼 Administrador Clínico

Encargado de la gestión macro de la institución. Opera en dashboard-admin.html.

  • Módulo de Dashboard: Análisis estadístico extrayendo datos de data.js (Pacientes totales, médicos activos, consultas finalizadas e ingresos financieros).
  • Módulo de Especialidades: CRUD (Crear, Leer, Actualizar, Eliminar) para el catálogo de especialidades que los pacientes verán al intentar agendar una cita.
  • Módulo de Doctores: Aprobación de cuentas de médicos, configuración de sus honorarios (tarifas) y vinculación a una especialidad.
  • Módulo Financiero (Pagos): Visualización de todas las transacciones realizadas por los pacientes en la pasarela de pagos.

2.2. 👨‍⚕️ Doctor (Especialista)

Profesional de la salud. Opera en dashboard-doctor.html y videoconsulta.html.

  • Bandeja de Solicitudes: Puede aceptar o rechazar solicitudes de citas "pendientes" enviadas por los pacientes. Al aceptar, el estado cambia a "confirmada".
  • Agenda Diaria: Visualiza un calendario de sus citas para el día en curso.
  • Historial Clínico: Tiene acceso de lectura a los antecedentes personales, quirúrgicos y enfermedades crónicas del paciente antes y durante la videoconsulta.
  • Emisión Médica: Al finalizar la cita, el módulo le obliga/permite generar Recetas Médicas, Certificados de Reposo y Órdenes de Laboratorio, los cuales se anexan al perfil del paciente.

2.3. 🧑 Paciente

El cliente final. Opera en dashboard-paciente.html y el Wizard de agendar-cita.html.

  • Agendamiento Asistido (Wizard): Proceso interactivo de 5 etapas (Especialidad, Doctor, Horario, Motivo, Pago) diseñado para maximizar la usabilidad.
  • Centro de Notificaciones: Recibe alertas (simuladas) cuando su médico aprueba o reprograma una cita.
  • Documentación Médica: Acceso a un repositorio de "Documentos Clínicos" donde puede descargar copias (previsualizaciones PDF simuladas) de las recetas emitidas por sus médicos.

4. Ingeniería de Flujos (Workflows)

4.1. El Módulo de Videoconsulta y WebRTC Simulado

El archivo videoconsulta.html es el núcleo técnico más complejo. No utiliza un SDK real de video (como Twilio o Agora) para mantener la naturaleza "Serverless", sino que simula la experiencia mediante manipulación avanzada del DOM y Media APIs.

🎥 Media Stream Simulator

Al hacer clic en "Unirse", el sistema activa la cámara web real del usuario mediante navigator.mediaDevices.getUserMedia() para el local stream, mientras que simula el remote stream (la otra persona) mediante estados visuales.

⏱️ Temporizador y Estados

Un worker/intervalo en JavaScript actualiza la duración de la llamada cada segundo. Si la conexión se "pierde" (simulado para UI), la UI se actualiza reactivamente sin recargar la página.

📝 Panel Lateral Médico

Durante la llamada, el sistema extrae el pacienteId de la URL, consulta a la base de datos local y despliega el "Historial Clínico" completo a la derecha, para que el médico no deba salir de la pantalla.

4.2. Asistente Inteligente (MediBot Chatbot)

Controlado por js/chatbot.js, MediBot es un widget flotante presente en el layout global.

  • Implementa un árbol de decisiones simple (Hardcoded NLP) basado en palabras clave.
  • Puede responder preguntas sobre horarios de atención, métodos de pago y especialidades leyendo directamente la tabla telemed_db.especialidades.
  • Regla de Negocio Crítica: Si detecta palabras relacionadas a síntomas graves ("dolor pecho", "emergencia"), cancela el flujo conversacional y despliega una alerta roja derivando al número de WhatsApp de emergencias de la clínica.

6. Despliegue y Arquitectura de Archivos

Directorio / Archivo Responsabilidad Técnica
/css/main.css Sistema de diseño (Design System). Contiene las variables CSS de colores (Primary, Secondary), tipografías, y la lógica del Dark Mode y Glassmorphism.
/js/auth.js Middleware de seguridad. Se carga en el <head> de cada archivo para verificar tokens antes de pintar el DOM. Maneja Modales y Toasts globales.
/js/data.js Capa DAO (Data Access Object). Funciones como initDB(), getDoctores(), actualizarCita(). Abstrae la lectura cruda de LocalStorage.
dashboard-*.html Vistas específicas. Cada una requiere cargar su respectivo script controlador (Ej. admin.js para dashboard-admin.html) que inyecta los datos en las tablas.
Estado de Producción: Dado que el proyecto carece de dependencias en Node (npm) o compiladores (Webpack/Vite), el proyecto se despliega simplemente sirviendo la carpeta estática. En Vercel, basta con conectar el repositorio de GitHub y dejar la configuración de Build en blanco.
